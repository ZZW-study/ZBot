## 二、核心类详解

### 2.1 AgentLoop（AI主循环引擎）⭐最重要

**文件位置**：`nanobot/agent/loop.py`

**职责**：项目的"大脑"，统筹LLM调用、工具执行、会话管理、消息处理。

#### 属性

| 属性名 | 类型 | 说明 |
|--------|------|------|
| `bus` | `MessageBus` | 消息总线，负责模块间通信 |
| `provider` | `LiteLLMProvider` | LLM调用器，负责与大模型API交互 |
| `tools` | `ToolRegistry` | 工具注册表，管理所有可用工具 |
| `context` | `ContextBuilder` | 上下文构建器，组装系统提示和消息 |
| `session_manager` | `SessionManager` | 会话管理器，持久化对话历史 |
| `subagent_manager` | `SubagentManager` | 子代理管理器，支持并行任务 |
| `workspace` | `Path` | 工作区路径 |
| `model` | `str` | 默认LLM模型 |
| `max_iterations` | `int` | 工具调用最大迭代次数 |
| `memory_window` | `int` | 记忆窗口大小 |
| `_running` | `bool` | 运行状态标志 |

#### 方法

| 方法名 | 参数 | 返回值 | 说明 |
|--------|------|--------|------|
| `__init__` | bus, provider, workspace, ... | None | 初始化所有组件 |
| `run` | - | None | **核心方法**：启动主循环，监听消息总线 |
| `stop` | - | None | 停止主循环 |
| `process_direct` | message, session_key, ... | str | **核心方法**：直接处理单条消息 |
| `_process_message` | session, messages, ... | str | 内部方法：处理消息的核心逻辑 |
| `_run_agent_loop` | messages, session, ... | str | 内部方法：LLM推理+工具执行循环 |
| `_save_turn` | session, messages, skip | None | 保存当前对话轮次 |
| `_consolidate_memory` | session, archive_all | bool | 合并/总结记忆 |
| `_register_default_tools` | - | None | 注册默认工具 |
| `_connect_mcp` | - | None | 连接MCP服务器 |
| `close_mcp` | - | None | 关闭MCP连接 |
| `_handle_stop` | msg | None | 处理停止命令 |
| `_dispatch` | msg | None | 分发消息到处理器 |
| `_set_tool_context` | channel, chat_id, ... | None | 设置工具上下文 |

### 2.2 LiteLLMProvider（LLM统一调用器）⭐核心

**文件位置**：`nanobot/providers/litellm_provider.py`

**职责**：统一封装所有大模型API调用，支持OpenAI/Anthropic/国内厂商/本地模型。

#### 属性

| 属性名 | 类型 | 说明 |
|--------|------|------|
| `default_model` | `str` | 默认模型名称 |
| `extra_headers` | `dict` | 额外请求头 |
| `_gateway` | `ProviderSpec \| None` | 网关配置（如OpenRouter） |
| `api_key` | `str \| None` | API密钥 |
| `api_base` | `str \| None` | API地址 |

#### 方法

| 方法名 | 参数 | 返回值 | 说明 |
|--------|------|--------|------|
| `__init__` | api_key, api_base, default_model, ... | None | 初始化提供商 |
| `chat` | messages, tools, model, ... | LLMResponse | **核心方法**：发送聊天请求 |
| `_resolve_model` | model | str | 模型名称标准化 |
| `_setup_env` | api_key, api_base, model | None | 设置环境变量 |
| `_supports_cache_control` | model | bool | 判断是否支持提示词缓存 |
| `_apply_cache_control` | messages, tools | tuple | 注入缓存控制标记 |
| `_sanitize_messages` | messages, extra_keys | list | 消息清洗（过滤非法字段） |
| `_normalize_tool_call_id` | tool_call_id | Any | 标准化工具调用ID |
| `_parse_response` | response | LLMResponse | 解析LLM响应 |
| `_apply_model_overrides` | model, kwargs | None | 应用模型专属参数 |
| `get_default_model` | - | str | 获取默认模型 |

## 三、数据类详解

### 3.1 Session（会话）

```python
@dataclass
class Session:
    key: str                              # 会话标识（如"cli:direct"）
    messages: list[dict[str, Any]]        # 对话消息列表
    created_at: datetime                  # 创建时间
    updated_at: datetime                  # 更新时间
    metadata: dict[str, Any]              # 元数据
    last_consolidated: int                # 已合并消息数
    
    def add_message(self, role, content, **kwargs): ...
    def get_history(self, max_messages=500): ...
    def clear(self): ...
```

---

### 3.2 LLMResponse（LLM响应）

```python
@dataclass
class LLMResponse:
    content: str | None                   # 文本内容
    tool_calls: list[ToolCallRequest]     # 工具调用请求
    finish_reason: str                    # 结束原因（stop/tool_calls）
    usage: dict[str, int]                 # Token统计
    reasoning_content: str | None         # 思考过程
    thinking_blocks: list[dict] | None    # Anthropic思考块
    
    def has_tool_calls(self) -> bool: ...
```

---

### 3.3 InboundMessage（入站消息）

```python
@dataclass
class InboundMessage:
    channel: str                          # 消息渠道（cli/whatsapp/telegram）
    sender_id: str                        # 发送者ID
    chat_id: str                          # 聊天ID
    content: str                          # 消息内容
    media: list[str] | None               # 媒体文件路径
    message_id: str | None                # 消息ID
    metadata: dict[str, Any]              # 元数据
    
    @property
    def session_key(self) -> str: ...     # 生成会话Key
```

---

### 3.4 OutboundMessage（出站消息）

```python
@dataclass
class OutboundMessage:
    channel: str                          # 目标渠道
    chat_id: str                          # 目标聊天ID
    content: str                          # 消息内容
    message_id: str | None                # 回复的消息ID
    metadata: dict[str, Any]              # 元数据
```

---

### 3.5 CronJob（定时任务）

```python
@dataclass
class CronJob:
    id: str                               # 任务ID
    name: str                             # 任务名称
    schedule: CronSchedule                # 执行计划
    payload: CronPayload                  # 执行内容
    enabled: bool                         # 是否启用
    created_at: datetime                  # 创建时间
    next_run: datetime | None             # 下次执行时间
    last_run: datetime | None             # 上次执行时间
```

---

### 3.6 ProviderSpec（提供商规格）

```python
@dataclass(frozen=True)
class ProviderSpec:
    name: str                             # 提供商名称
    label: str                            # 显示标签
    litellm_prefix: str | None            # LiteLLM前缀
    env_key: str | None                   # 环境变量名
    default_api_base: str | None          # 默认API地址
    is_oauth: bool                        # 是否OAuth认证
    is_local: bool                        # 是否本地模型
    is_gateway: bool                      # 是否网关模式
    supports_prompt_caching: bool         # 是否支持提示词缓存
    skip_prefixes: list[str]              # 跳过前缀的模型
    model_overrides: list[tuple]          # 模型专属参数覆盖
```

## 六、为什么这些类能协同工作？

### 6.1 核心设计原则

#### 原则1：依赖注入（Dependency Injection）

所有核心类都通过构造函数接收依赖对象：

```python
# AgentLoop接收所有依赖
agent = AgentLoop(
    bus=MessageBus(),                    # 消息总线
    provider=LiteLLMProvider(...),       # LLM调用器
    session_manager=SessionManager(...), # 会话管理器
    cron_service=CronService(...),       # 定时任务服务
)
```

**优势**：
- 解耦：各模块独立开发、测试
- 灵活：可替换任意组件（如换LLM提供商）
- 可测试：可注入Mock对象

---

#### 原则2：发布-订阅模式

MessageBus实现模块间解耦通信：

```python
# 发送方（CLI/通道）不需要知道接收方
await bus.publish_inbound(InboundMessage(...))

# 接收方（AgentLoop）不需要知道发送方
msg = await bus.consume_inbound()
```

**优势**：
- CLI、QQ、WhatsApp等通道只需发布消息
- AgentLoop只管消费消息，不关心来源
- 双方完全解耦，可独立扩展

---

#### 原则3：抽象基类（ABC）

所有可扩展组件都定义抽象基类：

```python
# 工具抽象基类
class Tool(ABC):
    @property
    @abstractmethod
    def name(self) -> str: ...
    
    @abstractmethod
    async def execute(self, **kwargs) -> str: ...

# 通道抽象基类
class BaseChannel(ABC):
    @abstractmethod
    async def start(self) -> None: ...
    
    @abstractmethod
    async def send(self, msg: OutboundMessage) -> None: ...

# LLM提供商抽象基类
class LLMProvider(ABC):
    @abstractmethod
    async def chat(self, messages, ...) -> LLMResponse: ...
```

**优势**：
- 强制子类实现关键方法
- 统一接口，多态调用
- 新增功能只需继承基类

---

#### 原则4：注册表模式

工具和提供商通过注册表动态管理：

```python
# 工具注册
tools.register(ReadFileTool(workspace))
tools.register(ExecTool(config))

# 提供商注册
PROVIDERS = [
    ProviderSpec(name="openai", ...),
    ProviderSpec(name="anthropic", ...),
]
```

**优势**：
- 动态增删组件
- 配置化扩展
- 无硬编码if-elif

## 七、总结：项目能顺畅运行的核心原因

### 1. 清晰的分层架构
- CLI入口层 → 网关服务层 → 核心业务层 → 通道适配层 → LLM提供商层
- 每层职责明确，单向依赖

### 2. 解耦的通信机制
- MessageBus实现发布-订阅模式
- 模块间无直接依赖，通过消息队列通信

### 3. 统一的抽象接口
- Tool、BaseChannel、LLMProvider抽象基类
- 约束子类行为，支持多态调用

### 4. 灵活的注册表设计
- 工具、提供商、通道都可动态注册
- 配置驱动的扩展机制

### 5. 完善的生命周期管理
- 启动顺序、运行监控、优雅关闭
- 资源正确释放，无内存泄漏

### 6. 异步并发设计
- asyncio实现高效并发
- 多任务并行执行（Agent + 通道 + 定时任务 + 心跳）
