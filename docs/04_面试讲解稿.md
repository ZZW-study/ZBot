# Nanobot 3分钟面试口述讲解稿

## 开场白（15秒）

> 面试官好，我来介绍我的Nanobot项目。这是一个**多通道AI智能体框架**，核心特点是支持多种大模型统一调用、多通道通信、内置工具系统，可以理解为一个"可扩展的AI助手开发平台"。

---

## 一、项目背景与定位（30秒）

**我为什么要做这个项目？**

当时我需要一个能同时支持多个聊天平台（CLI、WhatsApp、Telegram）的AI助手，但发现市面上的方案要么只能用一家大模型，要么不支持多通道。

**Nanobot的核心价值**：
1. **统一LLM调用**：一套代码兼容OpenAI、Anthropic、国内厂商、本地模型
2. **多通道支持**：同一套逻辑，可在CLI、WhatsApp、Telegram等平台运行
3. **工具扩展**：内置文件操作、Shell执行、定时任务、MCP协议
4. **长期记忆**：支持AI总结对话、存储重要信息

---

## 二、技术架构（45秒）

**整体架构分为5层**：

```
CLI入口层 → 网关服务层 → 核心业务层 → 通道适配层 → LLM提供商层
```

**核心模块**：

1. **AgentLoop**：AI主循环引擎，负责处理用户消息、调用LLM、执行工具
2. **LiteLLMProvider**：统一LLM调用器，通过注册表配置实现多厂商兼容
3. **ContextBuilder**：上下文构建器，组装系统提示、对话历史、用户消息
4. **MessageBus**：消息总线，实现模块间解耦通信
5. **ToolRegistry**：工具注册表，动态注册和管理所有工具

**设计模式**：
- 抽象工厂模式：Tool基类衍生多种工具
- 注册表模式：ProviderSpec驱动多厂商配置
- 发布-订阅模式：MessageBus实现模块解耦

---

## 三、核心难点与解决方案（60秒）

### 难点1：多LLM厂商兼容

**问题**：不同厂商API格式、参数、认证方式差异巨大。

**解决**：设计ProviderSpec注册表，通过配置驱动而非硬编码。

```python
# 示例：一个策略对象搞定一个厂商
ProviderSpec(
    name="anthropic",
    litellm_prefix="anthropic",
    env_key="ANTHROPIC_API_KEY",
    supports_prompt_caching=True,  # 支持提示词缓存
)
```

### 难点2：工具调用循环

**问题**：AI可能连续调用多个工具，甚至嵌套调用，如何控制？

**解决**：AgentLoop中的迭代循环设计：
- 设置最大迭代次数（max_iterations）防止死循环
- 每次工具结果都加入消息历史
- 检测LLM返回的finish_reason判断是否结束

### 难点3：上下文窗口管理

**问题**：对话越来越长，会超出LLM上下文限制。

**解决**：
1. **记忆窗口**：只保留最近N轮对话
2. **消息合并**：AI定期总结历史，写入MEMORY.md
3. **渐进式加载**：技能先给摘要，需要时再读完整文件

## 四、项目亮点（30秒）

1. **100%配置化**：新增LLM厂商只需加配置，无需改代码
2. **提示词缓存**：Anthropic/OpenRouter自动注入缓存控制，节省Token费用
3. **安全沙箱**：Shell工具内置危险命令过滤、路径越界检查
4. **多模态支持**：自动将图片转base64，支持多模态LLM

## 常见追问及回答

### Q1：为什么选择LiteLLM而不是自己封装？

**答**：LiteLLM已经处理了大部分厂商的兼容性问题，比如OpenAI格式转换、错误重试、流式响应。我在其基础上做了两层增强：
1. **注册表驱动**：通过ProviderSpec配置特殊逻辑
2. **消息清洗**：过滤非法字段、标准化工具ID

### Q2：工具调用如何保证安全性？

**答**：三层防护：
1. **参数校验**：根据JSON Schema验证所有参数
2. **白名单机制**：只允许配置的命令执行
3. **路径限制**：restrict_to_workspace模式下禁止越界访问

### Q3：会话数据如何持久化？

**答**：采用JSONL格式逐行存储：
- 第一行是元数据（会话ID、创建时间）
- 后续每行是消息记录
- 追加写入高效，支持大文件
- 内存缓存热点会话，减少磁盘IO

### Q4：如何扩展新的聊天通道？

**答**：继承BaseChannel，实现三个方法：
- `start()`：启动通道监听
- `send()`：发送消息
- `stop()`：停止通道

然后在注册表中声明即可，ChannelManager会自动发现和加载。
