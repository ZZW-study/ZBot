# 技能创建器

本技能用于指导你高效、规范地创建可用的智能体技能。

## 技能简介

技能是模块化、自包含的扩展包，用于为智能体补充专项知识、工作流与工具，相当于特定领域或任务的“上岗指南”。
它们能让通用智能体转变为具备专业流程知识的专用智能体，而这些流程知识通常无法直接由大模型完全掌握。

### 技能提供的能力

1. **专业工作流**：面向特定领域的多步骤执行流程
2. **工具集成**：针对特定文件格式或 API 的使用说明
3. **领域专业知识**：企业专属规则、数据结构、业务逻辑
4. **资源包**：用于复杂与重复任务的脚本、参考文档与资源文件

## 核心设计原则

### 简洁至上

上下文窗口是公共资源。技能会与系统提示、对话历史、其他技能元数据、用户真实请求共享上下文空间。

**默认前提：智能体本身已经足够聪明。**
只补充智能体原本不具备的上下文。对每一段信息都提出质疑：
“智能体真的需要这段说明吗？”“这段内容的价值是否对得起它占用的 token？”

优先使用简洁示例，避免冗长解释。

### 设定合理的自由度

根据任务的稳定性与变化程度，匹配对应的约束强度：

**高自由度（纯文本说明）**：适用于多种方案均合理、决策依赖上下文、依靠经验判断的场景。

**中等自由度（伪代码或带参数的脚本）**：适用于存在推荐方案、允许少量变化、行为可通过配置调整的场景。

**低自由度（固定脚本、极少参数）**：适用于操作脆弱易出错、一致性要求极高、必须严格按步骤执行的场景。

可以把智能体想象成在路径上行走：悬崖边的窄桥需要严格护栏（低自由度），开阔草地则允许多条路线（高自由度）。

### 技能组成结构

每个技能都包含一个必选的 `SKILL.md` 文件，以及可选的配套资源：

```
skill-name/
├── SKILL.md（必选）
│   ├── YAML 前置元数据（必选）
│   │   ├── name:（必选）
│   │   └── description:（必选）
│   └── Markdown 使用说明（必选）
└── 配套资源（可选）
    ├── scripts/          - 可执行代码（Python/Bash 等）
    ├── references/       - 按需加载到上下文的参考文档
    └── assets/           - 用于最终输出的文件（模板、图标、字体等）
```

#### SKILL.md（必选）

每个 `SKILL.md` 包含两部分：

- **前置元数据（YAML）**：包含 `name` 和 `description` 字段。
  这是智能体判断是否启用该技能的唯一依据，因此必须清晰、完整地说明技能用途与触发场景。
- **正文（Markdown）**：技能的使用说明与指引，仅在技能触发后才会加载（部分场景甚至不加载）。

#### 配套资源（可选）

##### 脚本（`scripts/`）

可直接执行的代码（Python/Bash 等），用于需要确定性结果或会被反复编写的任务。

- **适用场景**：相同代码被反复编写，或需要稳定、可复现的执行结果
- **示例**：`scripts/rotate_pdf.py` 用于 PDF 旋转任务
- **优势**：节省 token、执行结果确定、可不加载到上下文直接运行
- **注意**：脚本仍可能需要被智能体读取，用于修改或适配环境

##### 参考文档（`references/`）

用于在需要时加载到上下文、为智能体提供决策依据的文档与资料。

- **适用场景**：智能体执行任务时需要查阅的文档
- **示例**：`references/finance.md` 财务数据结构、`references/mnda.md` 公司保密协议模板、`references/policies.md` 公司制度、`references/api_docs.md` 接口规范
- **使用场景**：数据库结构、API 文档、领域知识、公司制度、详细工作流指南
- **优势**：保持 `SKILL.md` 精简，仅在智能体判断需要时加载
- **最佳实践**：若文件较大（>1 万字），可在 `SKILL.md` 中提供 grep 搜索方式
- **避免重复**：信息要么放在 `SKILL.md`，要么放在参考文档，不要重复。
  详细内容优先放入参考文档，只把核心流程与工作流留在 `SKILL.md` 中，避免挤占上下文。

##### 资源文件（`assets/`）

不加载到上下文、仅用于智能体生成最终输出时使用的文件。

- **适用场景**：技能需要在最终输出中使用固定文件
- **示例**：`assets/logo.png` 品牌素材、`assets/slides.pptx` PPT 模板、`assets/frontend-template/` HTML/React 脚手架、`assets/font.ttf` 字体文件
- **使用场景**：模板、图片、图标、脚手架代码、字体、待复制或修改的示例文档
- **优势**：将输出资源与说明文档分离，智能体可直接使用而不必加载进上下文

#### 技能中不应包含的内容

技能只应保留支撑其功能的核心文件，**禁止**添加无关文档或辅助文件，例如：

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- 等

技能只应包含智能体完成任务所需的信息，不应包含开发过程说明、安装测试步骤、面向用户的文档等。
多余文件只会造成混乱。

### 渐进式披露设计原则

技能采用三级加载机制，高效管理上下文：

1. **元数据（名称 + 描述）**：始终在上下文中（约 100 词）
2. **SKILL.md 正文**：技能触发时加载（＜5000 词）
3. **配套资源**：智能体按需加载（无上限，因为脚本可直接执行而不必读入上下文）

#### 渐进式披露模式

`SKILL.md` 正文只保留核心内容，控制在 500 行以内，避免上下文膨胀。
接近行数上限时，应将内容拆分到独立文件。
拆分后必须在 `SKILL.md` 中明确引用并说明何时使用，确保智能体能够发现并正确调用。

**核心原则**：如果一个技能支持多种场景、框架或选项，只把核心工作流与选择逻辑留在 `SKILL.md`，
将各场景专属细节（模式、示例、配置）放入独立参考文件。

**模式 1：总览指引 + 参考文件**

```markdown
# PDF 处理

## 快速开始

使用 pdfplumber 提取文本：
[代码示例]

## 高级功能

- **表单填写**：完整指南见 [FORMS.md](FORMS.md)
- **API 参考**：所有方法见 [REFERENCE.md](REFERENCE.md)
- **使用示例**：常见模式见 [EXAMPLES.md](EXAMPLES.md)
```

智能体只在需要时才加载 FORMS.md、REFERENCE.md 或 EXAMPLES.md。

**模式 2：按领域划分**

对包含多个领域的技能，按领域组织内容，避免加载无关信息：

```
bigquery-skill/
├── SKILL.md（总览与导航）
└── reference/
    ├── finance.md（收入、计费指标）
    ├── sales.md（商机、销售漏斗）
    ├── product.md（接口使用、功能）
    └── marketing.md（活动、归因）
```

当用户询问销售指标时，智能体只读取 sales.md。

同理，支持多种框架或平台的技能可按平台划分：

```
cloud-deploy/
├── SKILL.md（工作流 + 平台选择）
└── references/
    ├── aws.md（AWS 部署模式）
    ├── gcp.md（GCP 部署模式）
    └── azure.md（Azure 部署模式）
```

用户选择 AWS 后，智能体只读取 aws.md。

**模式 3：条件式详情**

展示基础内容，高级内容通过链接跳转：

```markdown
# DOCX 处理

## 创建文档

使用 docx-js 创建新文档，详见 [DOCX-JS.md](DOCX-JS.md)。

## 编辑文档

简单编辑可直接修改 XML。

**如需修订模式**：详见 [REDLINING.md](REDLINING.md)
**如需 OOXML 细节**：详见 [OOXML.md](OOXML.md)
```

智能体只在用户需要对应功能时才读取 REDLINING.md 或 OOXML.md。

**重要规范**：

- **避免深层嵌套引用**：参考文件只从 SKILL.md 直接跳转一级
- **长参考文件结构**：超过 100 行的文件应在顶部加目录，方便智能体预览全貌

## 技能创建流程

技能创建分为以下步骤：

1. 通过真实示例理解技能用途
2. 规划可复用的技能内容（脚本、参考文档、资源文件）
3. 初始化技能（运行 init_skill.py）
4. 编辑技能（实现资源并编写 SKILL.md）
5. 打包技能（运行 package_skill.py）
6. 根据实际使用迭代优化

请按顺序执行，除非明确某一步不适用才可跳过。

### 技能命名规则

- 只使用小写字母、数字和短横线；将用户输入的名称格式化为短横线格式（如 “Plan Mode” → `plan-mode`）
- 名称长度不超过 64 字符（字母、数字、短横线）
- 优先使用简短、以动词开头的短语
- 按工具做命名空间区分以提升触发准确性（如 `gh-address-comments`、`linear-address-issue`）
- 技能文件夹名称必须与技能名称完全一致

### 步骤 1：通过真实示例理解技能用途

仅当技能使用场景已非常清晰时才可跳过。即使是已有技能，这一步依然有价值。

要创建高效的技能，必须清晰掌握技能的真实使用案例。
这些案例可以来自用户真实需求，或经用户确认的模拟需求。

例如，在开发图片编辑技能时，可以询问：

- “图片编辑技能需要支持哪些功能？编辑、旋转，还是其他？”
- “可以举一些实际使用这个技能的例子吗？”
- “我能想到用户会说‘去掉图片红眼’‘旋转图片’，你预想中还有哪些用法？”
- “用户说什么话时应该触发这个技能？”

避免一次性提出过多问题，优先问最重要的，再逐步补充。

当你清晰掌握技能应支持的功能后，本步骤完成。

### 步骤 2：规划可复用的技能内容

将真实案例转化为规范技能，需要对每个案例进行分析：

1. 思考从零如何完成该任务
2. 识别在重复执行时能提高效率的脚本、参考文档与资源文件

示例：开发 `pdf-editor` 技能处理“旋转 PDF”类请求时：

1. 旋转 PDF 每次都要写相同代码
2. 可以在技能中内置 `scripts/rotate_pdf.py`

示例：开发 `frontend-webapp-builder` 技能处理“开发待办应用”“开发步数仪表盘”时：

1. 编写前端应用每次都需要相同的 HTML/React 脚手架
2. 可以在技能中内置 `assets/hello-world/` 模板项目

示例：开发 `big-query` 技能处理“今日登录用户数”类查询时：

1. 查询 BigQuery 每次都需要重新梳理表结构与关联关系
2. 可以在技能中内置 `references/schema.md` 数据表结构文档

通过分析每个真实案例，列出需要包含的可复用资源：scripts、references、assets。

### 步骤 3：初始化技能

至此可以正式创建技能。

仅当技能已存在、只需迭代或打包时跳过，直接进入下一步。

从零创建新技能时，**务必运行 `init_skill.py`**。
该脚本会自动生成标准模板目录，包含技能所需的全部基础结构，让创建过程更高效、更规范。

在 nanobot 中，自定义技能应放在工作区的 `skills/` 目录下，以便运行时自动发现，例如：
`<workspace>/skills/my-skill/SKILL.md`

使用方式：

```bash
scripts/init_skill.py <skill-name> --path <output-directory> [--resources scripts,references,assets] [--examples]
```

示例：

```bash
scripts/init_skill.py my-skill --path ./workspace/skills
scripts/init_skill.py my-skill --path ./workspace/skills --resources scripts,references
scripts/init_skill.py my-skill --path ./workspace/skills --resources scripts --examples
```

脚本功能：

- 在指定路径创建技能目录
- 生成带标准前置元数据与 TODO 占位符的 SKILL.md 模板
- 根据 `--resources` 可选创建资源目录
- 设置 `--examples` 时会自动添加示例文件

初始化后，自定义 SKILL.md 并添加所需资源。
如果使用了 `--examples`，请替换或删除不需要的占位文件。

### 步骤 4：编辑技能

编辑新生成或已有的技能时，牢记：技能是给另一个智能体实例使用的。
只加入对智能体有用、非显而易见的信息，思考哪些流程知识、领域细节、可复用资源能帮助它更高效地完成任务。

#### 学习成熟设计模式

根据技能需求参考以下指南：

- **多步骤流程**：详见 `references/workflows.md` 顺序工作流与条件逻辑
- **特定输出格式或质量标准**：详见 `references/output-patterns.md` 模板与示例模式

这些文件包含经过验证的技能设计最佳实践。

#### 先实现可复用资源

开始开发时，优先完成前面规划的可复用资源：`scripts/`、`references/`、`assets/`。
这一步可能需要用户提供素材。
例如开发 `brand-guidelines` 技能时，用户可能需要提供品牌资源或模板放入 `assets/`，或提供文档放入 `references/`。

新增脚本必须实际运行测试，确保无 bug 且输出符合预期。
如果同类脚本较多，只需抽样测试以保证整体可用并平衡开发时间。

如果使用了 `--examples`，删除技能不需要的占位文件。只创建实际需要的资源目录。

#### 更新 SKILL.md

**写作规范**：统一使用祈使/不定式句式。

##### 前置元数据

编写包含 `name` 和 `description` 的 YAML 前置配置：

- `name`：技能名称
- `description`：技能的核心触发依据，帮助智能体判断何时使用
  - 同时说明技能功能与具体触发场景
  - 所有“何时使用”的信息必须写在这里，而不是正文中
    正文只在触发后加载，正文中的使用说明对智能体触发无帮助
  - docx 技能示例描述：
    “用于专业文档（.docx）的创建、编辑与分析，支持修订模式、批注、格式保留与文本提取。
    当智能体需要处理专业文档时使用，包括：(1) 新建文档 (2) 修改编辑内容 (3) 使用修订模式 (4) 添加批注，以及其他文档相关任务”

保持前置元数据精简。在 nanobot 中，`metadata` 和 `always` 按需使用，非必要不添加额外字段。

##### 正文

编写技能及其配套资源的使用说明。

### 步骤 5：打包技能

技能开发完成后，需要打包为可分发的 `.skill` 文件交付使用。
打包过程会先自动校验技能是否符合规范：

```bash
scripts/package_skill.py <path/to/skill-folder>
```

可指定输出目录：

```bash
scripts/package_skill.py <path/to/skill-folder> ./dist
```

打包脚本会：

1. **自动校验技能**，检查内容包括：
   - YAML 前置格式与必填字段
   - 技能命名规范与目录结构
   - 描述完整性与质量
   - 文件组织与资源引用合法性

2. **校验通过后打包**，生成以技能名命名的 `.skill` 文件（如 `my-skill.skill`），
   包含所有文件并保留目录结构。`.skill` 本质是改后缀的 ZIP 包。

安全限制：符号链接会被拒绝，存在符号链接时打包直接失败。

如果校验失败，脚本会报错并退出，不生成包。修复问题后重新运行打包命令。

### 步骤 6：迭代优化

技能测试后，用户可能会提出改进需求。通常在刚使用完、对表现记忆清晰时反馈最有价值。

**迭代流程：**

1. 在真实任务中使用技能
2. 发现卡点或低效问题
3. 确定需要更新 SKILL.md 还是配套资源
4. 实施修改并再次测试
