# ZBot 项目审查

> 范围：基于当前仓库代码与 `uv run` 运行路径进行审查。仅列潜在错误与改进方向，不涉及实现细节。

## 高优先级

### 1. 配置容错分支会直接崩溃
- 位置：`ZBot/config/loader.py:21-43`
- 风险：当配置文件不存在、损坏或校验失败时，系统没有真正回退到默认配置，而是进入异常后的二次崩溃，启动容错路径失效。
- 短期方向：把“缺配置 / 坏配置 / 校验失败”统一收敛为可继续启动的稳定结果。
- 长期方向：建立配置加载的单一失败语义，让默认值、磁盘值、环境值的优先级和降级行为保持一致。

### 2. 运行时依赖闭包与导入边界同时失效
- 位置：`pyproject.toml:7-20`，`ZBot/agent/__init__.py:11-23`，`ZBot/agent/context.py:13`，`ZBot/cron/service.py:26`
- 风险：在干净的 Windows / `uv run` 环境下，仅导入 `ZBot.agent` 就会因时区依赖缺口而失败，说明发布依赖与包导出边界都不稳定。
- 短期方向：保证项目声明的依赖足以支撑默认导入路径，并避免非必要运行时组件在包导入阶段提前执行。
- 长期方向：把“干净环境可导入、可启动、可执行核心命令”作为发布基线，持续收紧导入面的耦合。

### 3. Cron 表达式能力在项目环境中不可用
- 位置：`pyproject.toml:7-20`，`ZBot/cron/service.py:46-56`，`ZBot/cron/service.py:94-104`
- 风险：代码实现依赖 cron 表达式解析能力，但项目依赖声明没有覆盖，导致定时任务功能在实际运行环境中出现能力缺口。
- 短期方向：让调度能力与依赖声明保持一致，避免“代码支持、环境不支持”的落差。
- 长期方向：将功能开关、依赖声明和启动自检统一到同一套能力模型中。

### 4. 当前消息相关技能实际上不会进入 system prompt
- 位置：`ZBot/agent/context.py:50-74`，`ZBot/agent/context.py:210-226`
- 风险：`build_system_prompt()` 预留了按当前消息加载相关技能的分支，但主流程调用时没有传入 `current_message`，这条能力实际上失效。
- 短期方向：确保主消息构建链路与技能选择逻辑一致，避免“接口已设计、主流程未接入”的空转状态。
- 长期方向：把上下文拼装改成可验证的组合契约，减少隐藏分支长期失活。

### 5. 运行时元信息被持久化进会话历史
- 位置：`ZBot/agent/context.py:36-37`，`ZBot/agent/context.py:219-222`，`ZBot/agent/loop.py:559-610`，`ZBot/session/manager.py:64-84`
- 风险：当前时间等临时元信息会被原样写入 session，并在后续历史回放时继续进入模型上下文，造成历史污染和长期记忆噪声。
- 短期方向：保证运行时元信息只服务于当前轮推理，不进入持久化历史。
- 长期方向：明确区分“推理期上下文”和“会话期事实”，建立更清晰的上下文生命周期边界。

## 中优先级

### 6. 工作区级引导文件不会生效
- 位置：`ZBot/agent/context.py:32`，`ZBot/agent/context.py:369-372`
- 风险：上下文构建声明了 `AGENTS.md` / `SOUL.md` / `USER.md` / `TOOLS.md` / `MEMORY.md` 等引导文件，但实际只读取包内模板，不读取工作区内容，项目级规则无法真正接入。
- 短期方向：让引导来源与实际产品语义保持一致，避免“可配置”外观下的固定行为。
- 长期方向：把模板、工作区约束、用户定制拆分成清晰的层级化上下文来源。

### 7. 工具错误协议不一致，部分失败路径绕过上层语义
- 位置：`ZBot/agent/tools/registry.py:61-64`，`ZBot/agent/tools/web.py:148-149`，`ZBot/agent/tools/web.py:218-221`
- 风险：`ToolRegistry` 只识别以“错误：”开头的工具失败结果，但 `web` 工具在部分异常路径返回 `Error:`，导致上层重试提示和错误处理语义不一致。
- 短期方向：统一工具错误输出契约，避免同类失败在上层呈现出不同语义。
- 长期方向：建立结构化工具结果模型，减少依赖字符串前缀驱动流程判断。

### 8. Provider 内部契约不严格，存在空值渗透
- 位置：`ZBot/providers/litellm_provider.py:56-77`
- 风险：模型解析与缓存能力判断在未命中提供商时会返回空值而不是稳定类型，后续链路会放大这种不确定性。
- 短期方向：收紧 provider 层返回契约，让失败和不支持状态明确化。
- 长期方向：把模型路由、能力声明、参数兼容性收敛为显式能力矩阵，而不是隐式分支结果。

## 低优先级

### 9. Shell 工具存在解释器告警
- 位置：`ZBot/agent/tools/shell.py:273`
- 风险：当前存在无效转义序列告警，短期不会阻断运行，但会削弱零告警基线，也会增加未来版本兼容性的不确定性。
- 短期方向：清理解释器层面的告警，保持基础模块可在严格模式下通过。
- 长期方向：把“无告警编译/检查”纳入常规质量门槛，避免小问题积累成维护成本。
